title: /investigate
description: '障害または疑わしいリソースに対する根本原因調査を開始します — 初期バンドルを収集し、エージェントに説明します。'
---

`/investigate` スキルは、進行中の問題のトリアージを支援するエージェントを呼び込む方法です。スキルを使用すると、コンテキストとして使用するデータの初期バンドル（例：スペック、現在および以前のコンテナステータス、問題のあるリソースのイベント）を収集し、利用可能なものと調査のアプローチ方法についてエージェントに説明します。

その後、自然言語でフォローアップの質問ができ、エージェントはすでに持っているものから回答するか、新しいデータを取得するか、他のツールを使用するかを決定します。

```text
/investigate                             # prompts for a target
/investigate pod/checkout-7c9            # explicit kind/name
/investigate the api deployment          # natural-language target
/investigate why is checkout crashing    # natural-language + intent
```

自然言語ターゲットとスコープ（Namespace、ラベルセレクター、ワークロード名）がサポートされています（[概要](/ja/reference/skills/overview/)を参照）。

---

## 収集内容

:::note[初期バンドル]
- 問題のあるリソースのスペックとステータス
- 問題のあるリソースとそのオーナーのイベント（`Pod` の `ReplicaSet` と `Deployment`、`Job` の `CronJob` など）
- 現在のコンテナのログ、および存在する場合は以前のもの（`--previous`）— 障害を含む可能性が最も高い行に省略
- 明らかな関連リソース：バッキング `Service`、マウントされた `ConfigMap` と `Secret`（名前のみ、内容は含まない）、バインドされた `PVC`、参照された `ServiceAccount`
- 関連する場合、問題のあるPodがスケジュールされているNode — コンディション、capacity、および圧迫下にあるかどうか
:::

ソース：Kubernetes APIのみ。

---

## エージェントへの指示

スキルは調査へのアプローチ方法についてエージェントに指示します。

- データバンドルの読み取り方と、どのシグナルがどこを指す傾向があるか（終了コード、イベントの理由、一般的な状態の組み合わせ）。
- 現在の状態を必要とするフォローアップは、（今や古くなっている可能性がある）バンドルから推論するのではなく再取得すること。
- 状況が継続的な観察を必要とする場合は、スコープを絞ったイベント監視が利用可能なこと。
- 隣接スキルへの引き継ぎタイミング：ライブテールは [`/logs`](/ja/reference/skills/logs/)、シェルは [`/exec`](/ja/reference/skills/exec/)、時間経過のリソース使用量は [`/metrics`](/ja/reference/skills/metrics/)。

そのコンテキストが読み込まれると、セッションの残りは通常の会話です — 問題がどのような形であれ、聞きたいことを聞いてください。

---

## オプション

`/investigate` にはスキル固有のフラグはありません。ログ、時間ウィンドウ、リソースのスコープはプロンプトまたはフォローアップで自然言語を使用して指定してください。

[概要](/ja/reference/skills/overview/)のグローバルフラグが適用されます。
